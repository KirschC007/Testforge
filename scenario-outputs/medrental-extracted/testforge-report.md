## Ambiguity Gate

### B-005 — ⚠ Reduces Confidence
**Problem:** It's unclear if userId, clinicId, role in JWT are PII or sensitive data that needs special protection (e.g., encryption at rest for clinicId if it directly identifies a clinic).
**Question:** Are userId, clinicId, and role considered PII or sensitive data requiring specific protection measures beyond standard JWT security?

### B-010 — ⚠ Reduces Confidence
**Problem:** The spec says technician can 'view rentals' but GET /api/rentals says 'technician: device-focused view'. The exact scope of 'view rentals' for technician is not fully clear.
**Question:** What specific rental fields and filters are available to the technician role when viewing rentals, beyond just a 'device-focused view'?

### B-013 — ⚠ Reduces Confidence
**Problem:** The spec states 'nurse: cannot modify pricing' but doesn't explicitly state what happens if a nurse attempts to modify pricing (e.g., HTTP 403 or if the system simply ignores the input).
**Question:** What is the exact system response when a nurse attempts to modify pricing? Is it a 403, or is the pricing field simply ignored/reset?

### B-017 — ⚠ Reduces Confidence
**Problem:** The spec states 'billing: cannot access medical records' but doesn't define what constitutes 'medical records' in the system. Is it a specific resource, or any field with 'medical' in its name?
**Question:** What are the specific data fields or resources categorized as 'medical records' that the billing role cannot access?

### B-021 — ⚠ Reduces Confidence
**Problem:** The spec states 'admin: pricing' but doesn't specify which pricing fields or operations (e.g., setting dailyRate, modifying purchasePrice, adjusting late fees) are included in 'managing pricing'.
**Question:** What are the specific 'pricing' attributes and operations that the admin role can manage?

### B-034 — ⚠ Reduces Confidence
**Problem:** The spec refers to 'field visibility per role as above' for GET /api/devices/:id, but the 'above' section only describes field visibility for GET /api/devices (list). It's unclear if the exact same field restrictions apply to a single device detail view.
**Question:** Are the exact same field visibility rules (e.g., nurse sees no pricing, billing sees no maintenance) applied to GET /api/devices/:id as they are to GET /api/devices?

### B-059 — ⚠ Reduces Confidence
**Problem:** The spec states 'billing: all rentals with financial data' but doesn't explicitly define what constitutes 'financial data' for a rental.
**Question:** What specific fields are included in 'financial data' for rentals that the billing role can see?

### B-060 — ⚠ Reduces Confidence
**Problem:** The spec states 'technician: device-focused view' for listing rentals, but doesn't specify what this view entails (e.g., what fields are visible, what filters are available, or if it's a different endpoint).
**Question:** What specific data fields, filters, and display logic are part of the 'device-focused view' for technicians when listing rentals?

### B-090 — ⚠ Reduces Confidence
**Problem:** The spec says '→maintenance: set maintenanceStartDate' but does not specify the value. Is it the current date, or the date the maintenance is scheduled to begin?
**Question:** When a device transitions to 'maintenance' status, what value is assigned to 'maintenanceStartDate'?

### B-094 — ⚠ Reduces Confidence
**Problem:** The spec says '→decommissioned: ..., decommissionedReason' but does not specify if this reason is automatically populated from the input 'reason' field of the PATCH /api/devices/:id/status endpoint, or if it's a separate field.
**Question:** Is the 'decommissionedReason' automatically set from the 'reason' field provided in the PATCH /api/devices/:id/status call, or is it a separate field that needs to be explicitly set?

### B-101 — ⚠ Reduces Confidence
**Problem:** The spec states 'active→cancelled (admin only, with reason)' but doesn't specify if the reason is a required input for the cancellation or just a record.
**Question:** Is the 'reason' field a mandatory input when an admin cancels an active rental?

### B-107 — ⚠ Reduces Confidence
**Problem:** The spec says '→overdue: send overdue notification' but does not specify the recipient, content, or mechanism of this notification.
**Question:** Who receives the overdue notification, what information does it contain, and how is it sent (e.g., email, in-app)?

### B-113 — ⚠ Reduces Confidence
**Problem:** The spec says '→cancelled: device.status→available' but doesn't specify if this applies to all cancellation scenarios (e.g., if a device was cancelled after its startDate, should it still become available immediately?).
**Question:** Does the device always become 'available' immediately upon rental cancellation, regardless of the cancellation policy (e.g., after startDate)?

### B-113 — ⚠ Reduces Confidence
**Problem:** The spec says '→cancelled: ..., refund deposit if applicable' but the cancellation policy specifies different refund percentages. It's ambiguous when the refund is processed.
**Question:** Is the deposit refund processed immediately upon cancellation, or is it a separate process triggered by the cancellation?
