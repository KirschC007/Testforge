Kapitel 1: Einführung
1.1 Systemzweck
hey-listen.ai ist ein Multi-Tenant SaaS-Reservierungssystem für Restaurants. Das System
ermöglicht die vollständige Verwaltung von Tischreservierungen über mehrere Kanäle und bietet
KI-gestützte Funktionen zur Gästebindung und Risikominimierung.
Funktion

Beschreibung

Buchungsverwaltung

Erstellen, Ändern, Stornieren von Reservierungen über alle Kanäle

Multi-Channel

Widget (Web/iframe), Voice-Bot (Telefon-KI), Manuell (Staff/Admin)

Gäste-CRM

Besuchshistorie, Tags, Stammgast-Erkennung, VIP-Status, Sperrung

Warteliste

Automatische Benachrichtigung bei Verfügbarkeit

Rebooking Engine

KI-gestützte Reaktivierung inaktiver Stammgäste via SMS

Stripe Pre-Auth

Kreditkarten-Autorisierung bei KI-erkanntem No-Show-Risiko

Multi-Tenant

Vollständig isolierte Restaurants mit eigenem Admin-Bereich

DSGVO-Konformität

Automatische Datenlöschung, Consent-Management, Audit-Logs

Self-Healing

KI-gestützte automatische Fehlerbehebung via Sentry + LLM

2.2 Primärschlüssel-Format
Alle Tabellen verwenden INT UNSIGNED AUTO_INCREMENT . UUIDs werden nur für logische
Gruppierer verwendet: groupBookingId und seriesId in reservations (VARCHAR(36),
generiert mit nanoid() ).

2.3 Multi-Tenancy
Alle Tabellen mit Restaurant-Bezug haben restaurantId . Jede tRPC-Prozedur prüft ob der
authentifizierte User zu diesem Restaurant gehört. Super-Admin hat Zugriff auf alle Restaurants.

2.4 Authentifizierung
Restaurant-Admin / Staff: JWT in HttpOnly-Cookie ( hey_listen_session ). Ablauf: 24h. IdleTimeout: 4h. Parallele Sessions: max. 3 pro User.
Super-Admin (v7-Korrektur — Manus OAuth entfernt): Eigene Passwort-Authentifizierung —
identisch zu Restaurant-Admin/Staff. Eintrag in users -Tabelle mit role = 'super_admin' ,
restaurantId = NULL . Login über /login mit username + password (bcrypt, 12 Rounds). JWT

in HttpOnly-Cookie ( hey_listen_session ). Ablauf: 24h. Idle-Timeout: 30 Minuten (kürzer als
Admin/Staff wegen erhöhter Privilegien). Parallele Sessions: max. 3.

Begründung: Manus OAuth war eine Plattform-spezifische Abhängigkeit. Das System ist
vollständig eigenständig und benötigt keinen externen OAuth-Provider. Die eigene Passwort-

Auth ist bereits für alle anderen Rollen implementiert und bewährt.
Initiales Super-Admin-Konto: Wird beim ersten Server-Start automatisch angelegt wenn kein
super_admin -User

existiert.

API-Keys: bcrypt-Hash in api_keys -Tabelle. Prefix (10 Zeichen) im Klartext für Anzeige.
Gäste (Self-Service): 64-Zeichen-Hex-Token in reservations.selfServiceToken . Gültig bis

4.3 Die 9 Validierungsschritte (in dieser Reihenfolge)
Jede Buchung — egal aus welcher Quelle — durchläuft exakt diese 9 Schritte. Bei Fehler in einem
Schritt wird sofort abgebrochen und der Fehlercode zurückgegeben. Kein Schritt wird
übersprungen.
Schritt 1: Grundfeld-Validierung
Geprüft wird:
guestName : Nicht leer, min. 2 Zeichen, max. 255 Zeichen

guestPhone : Nicht leer, E.164-Format nach Normalisierung (Leerzeichen, Bindestriche,

Klammern werden entfernt), min. 7 Ziffern, max. 15 Ziffern

Kapitel 7: Öffnungszeiten, Sondertage, Pausen, TimeBlocks
7.1 Prioritäts-Hierarchie
Bei der Öffnungszeiten-Prüfung gilt folgende Priorität (höher = überschreibt):
1. time_blocks (höchste Priorität — blockiert immer)
2. exceptions (Feiertage/Sonderöffnungen)
3. opening_hours (reguläre Wochenzeiten)

7.2 Mitternacht-Buchungen
Buchungen die über Mitternacht gehen (z.B. 23:00 bis 01:00) sind möglich wenn:
closeTime ist kleiner als openTime (z.B. openTime=17:00, closeTime=02:00)

Das System erkennt dies automatisch: wenn closeTime < openTime , dann gilt closeTime
als nächster Tag
Die Buchung endTime kann auf den nächsten Tag fallen
Buchungsvalidierung prüft korrekt über Mitternacht

7.3 Pausen
Pausen werden in opening_hours.pauseStart und opening_hours.pauseEnd gespeichert.
Buchungen die in die Pause fallen werden abgelehnt. Buchungen die VOR der Pause beginnen

und NACH der Pause enden werden ebenfalls abgelehnt (die Buchungsdauer würde die Pause
überspannen).

7.4 TimeBlocks
TimeBlocks blockieren Buchungen für einen bestimmten Zeitraum:
affectsOnline = 1 : Blockiert Widget und Voice-Bot
affectsManual = 1 : Blockiert auch Staff und Admin

Beide können gleichzeitig aktiv sein
TimeBlocks können sich überschneiden (mehrere gleichzeitig aktiv)

7.5 Sondertage (Exceptions)
type = 'closed' : Restaurant geschlossen, alle Buchungen werden abgelehnt
type = 'special' : Andere Öffnungszeiten als regulär (z.B. Silvester nur bis 20:00)

Exceptions überschreiben opening_hours für diesen Tag
Bestehende Buchungen an einem nachträglich als geschlossen markierten Tag werden
NICHT automatisch storniert. Admin wird informiert und muss manuell handeln.

Wenn ein Gast im Widget buchen möchte und NO_TABLE_AVAILABLE zurückkommt:
1. Widget zeigt Alternativslots (max. 5)
2. Wenn keine Alternativslots akzeptiert werden: Widget zeigt “Warteliste”-Button (nur wenn
waitlistEnabled = 1 )

3. Gast gibt Name, Telefon, E-Mail (optional), Personenanzahl, gewünschte Zeit ein
4. Gast gibt SMS-Consent (Pflicht für Warteliste)
5. System prüft: waitlistMaxEntries für diesen Tag noch nicht erreicht
6. Wenn Limit erreicht: Fehler WAITLIST_FULL (DE: “Die Warteliste für diesen Tag ist voll.”, EN:
“The waitlist for this day is full.”)
7. Erstelle waitlist -Eintrag mit status = 'waiting' , expiresAt = NOW() + 72h
8. Sende Bestätigungs-SMS: “Sie stehen auf der Warteliste für {date} um {time}. Wir
benachrichtigen Sie wenn ein Platz frei wird.”

10.3 Benachrichtigung bei Verfügbarkeit
Wenn eine Buchung storniert wird (status → cancelled):
1. System prüft ob waitlistEnabled = 1 für dieses Restaurant
2. Suche Wartelisten-Einträge für das gleiche Datum mit status = 'waiting'
3. Wenn aiWaitlistEnabled = 1 : KI priorisiert Einträge nach:
Personenanzahl passt zum freigewordenen Tisch (höchste Priorität)
Wartezeit (länger wartet = höhere Priorität)
Stammgast-Status (Stammgäste bevorzugt)
Gewünschte Zeit (näher an gewünschter Zeit = höhere Priorität)
4. Wenn aiWaitlistEnabled = 0 : FIFO (Reihenfolge der Eintragung)
5. Ersten passenden Eintrag auswählen
6. Setze waitlist.status = 'notified' , notifiedAt = NOW()
7. Sende SMS mit Buchungslink: “Ein Platz ist frei! Buchen Sie jetzt: {link}” — Link gültig für
waitlistLinkMinutes Minuten

8. Der Link führt zum Widget mit vorausgefüllten Daten (Datum, Zeit, Personenanzahl)
9. Wenn Gast nicht innerhalb von waitlistLinkMinutes bucht: status = 'expired' ,
nächster Eintrag wird benachrichtigt

10.4 Slot-Blockierung
Wenn ein Wartelisten-Eintrag benachrichtigt wird ( status = 'notified' ):
Der freigewordene Tisch wird für waitlistLinkMinutes Minuten für andere WidgetBuchungen NICHT gesperrt
Andere Gäste können den Tisch weiterhin buchen
Wenn ein anderer Gast zuerst bucht: Wartelisten-Gast bekommt Fehler
NO_TABLE_AVAILABLE beim Klick auf den Link

System benachrichtigt dann automatisch den nächsten Wartelisten-Eintrag
Begründung: Eine Sperrung würde zu Problemen führen wenn der Wartelisten-Gast nicht
reagiert. First-come-first-served ist fairer.

10.5 LLM-Fallback
Wenn aiWaitlistEnabled = 1 und der LLM-Service nicht erreichbar ist:
System fällt automatisch auf FIFO zurück

Kein Fehler für den Gast
Eintrag in error_logs

10.6 Maximale Einträge
waitlistMaxEntries gilt pro Tag (nicht pro Zeitslot). Wenn das Limit erreicht ist, werden keine

neuen Einträge angenommen. Das Limit kann vom Admin zwischen 1 und 500 eingestellt werden
(Default: 50).

10.7 DSGVO-Ablauf
expiresAt = createdAt + 72h . Cron-Job waitlistExpiry läuft täglich und setzt status =
'expired' für alle Einträge wo expiresAt < NOW() und status = 'waiting' . Die Einträge

1. Findet Buchungen: status = 'seated' AND endTime + INTERVAL 30 MINUTE < NOW()
2. Setzt status = 'completed'
3. Erhöht guests.visitCount um 1
4. Setzt guests.isStammgast = 1 wenn visitCount >= 5

19.11 webhookRetry Job (v8: neu)
Läuft alle 5 Minuten.
Versucht fehlgeschlagene ausgehende Webhooks erneut zu senden.
Ablauf:

1. Findet Einträge in webhooks -Tabelle mit status = 'failed' AND nextRetryAt <= NOW()
AND attempts < maxAttempts

2. Sendet Webhook erneut (HMAC-signiert)
3. Bei Erfolg: status = 'sent'
4. Bei Fehler: attempts++ , nextRetryAt = NOW() + exponential_backoff
5. Nach maxAttempts (5): status = 'dead'

19.12 stripePreAuthCheck Job (v8: neu)
Läuft alle 5 Minuten.
Prüft ablaufende Stripe Pre-Auth-Autorisierungen und verlängert sie wenn nötig.
Ablauf:
1. Findet Buchungen: stripeAuthStatus = 'authorized' AND stripePreAuthExpiresAt <
NOW() + INTERVAL 48 HOUR

2. Verlängert Autorisierung via Stripe API (cancel + neue PaymentIntent)
3. Aktualisiert stripePaymentIntentId , stripePreAuthExpiresAt
4. Bei Fehler: stripeAuthStatus = 'capture_failed' , Admin-Alert

19.13 sessionCleanup Job (v8: neu)
Läuft stündlich.
Bereinigt abgelaufene User-Sessions aus der user_sessions -Tabelle.
Ablauf:
1. DELETE FROM user_sessions WHERE expiresAt < NOW()
2. Protokolliert Anzahl gelöschter Sessions in cron_job_logs

19.14 riskScoring Job (v8: neu)
Läuft täglich um 03:30 Uhr (UTC).
Aktualisiert Risiko-Scores aller Gäste die in den letzten 30 Tagen eine Buchung hatten.
Ablauf:
1. Findet Gäste mit aktiver Buchung in den letzten 30 Tagen

2. Berechnet Risiko-Score via LLM (anonymisierte Eingabe: noShowCount, visitCount,
lastVisit, cancellationRate)
3. Speichert Score in guests.riskScore

auditLogRetentionDays DAY

21.3 Sicherheit
HMAC-SHA256 Signatur:

X-HeyListen-Signature: sha256={hash}
X-HeyListen-Timestamp: {unix-timestamp}

Validierung:
1. Timestamp nicht älter als 5 Minuten (Replay-Schutz)
2. HMAC-SHA256 des Bodys mit Secret berechnen
3. Mit timingSafeEqual vergleichen
Replay-Attack (Edge Case EC-86): Angreifer replayed alten Webhook → Timestamp-Check (max.
5 Min alt) + processed_webhooks Idempotency (72h TTL).

21.4 Retry-Logik
Exponentieller Retry (5 Versuche):
Versuch 1: Sofort
Versuch 2: +1 Sekunde
Versuch 3: +10 Sekunden
Versuch 4: +60 Sekunden
Versuch 5: +5 Minuten
Nach 5 Fehlern: Webhook wird deaktiviert ( isActive = 0 ). Admin-Alert: “Webhook [URL]
wurde nach 5 Fehlern deaktiviert.”

32.3 Ebene 2: Logger-Redaction
Alle Log-Einträge werden durch einen Redaction-Filter geleitet, bevor sie in error_logs oder
Sentry gespeichert werden.
Redaction-Konfiguration:

// server/_core/logger.ts
const REDACTED_FIELDS = ['name', 'phone', 'email', 'notes', 'to', 'recipient',
'address'];
const REDACTED_PATTERNS = [
/\+\d{10,15}/g,

// Telefonnummern (E.164)

/[\w.-]+@[\w.-]+\.\w+/g,

// Email-Adressen

];
export function redact(obj: unknown): unknown {
if (typeof obj === 'string') {
return REDACTED_PATTERNS.reduce((s, p) => s.replace(p, '[REDACTED]'), obj);
}
if (typeof obj === 'object' && obj !== null) {
return Object.fromEntries(
Object.entries(obj).map(([k, v]) =>
REDACTED_FIELDS.includes(k.toLowerCase())
? [k, '[REDACTED]']
: [k, redact(v)]
)
);
}
return obj;
}

Anwendung: Jeder logger.error() , logger.warn() , logger.info() Aufruf läuft durch
redact() .

payment_intent.amount_capturable_updated

Tritt auf wenn Stripe-Autorisierung erfolgreich ist
Relevante Felder: payment_intent.id , payment_intent.amount ,
payment_intent.metadata.reservationId

Aktion:
1. reservations.stripePaymentIntentId = payment_intent.id
2. reservations.stripeAuthStatus = 'authorized'
3. reservations.status = 'confirmed'
4. Bestätigungs-SMS senden (wenn noch nicht gesendet)
5. Audit-Log: action = 'stripe_authorized'
payment_intent.canceled

Tritt auf wenn Autorisierung abläuft oder manuell storniert wird
Aktion:
1. reservations.stripeAuthStatus = 'cancelled'
2. Wenn reservations.status = 'pending_authorization' : status = 'cancelled'
3. Tisch freigeben
4. Storno-SMS senden
5. Audit-Log: action = 'stripe_auth_cancelled'

payment_intent.succeeded

Tritt auf wenn Capture erfolgreich ist (nach No-Show)
Aktion:
1. reservations.stripeAuthStatus = 'captured'
2. reservations.stripeCapturedAt = jetzt
3. Audit-Log: action = 'stripe_captured'
payment_intent.payment_failed

Tritt auf wenn Capture fehlschlägt
Aktion:
1. reservations.stripeAuthStatus = 'capture_failed'
2. Admin-Alert: “Stripe Capture fehlgeschlagen für Buchung [ID]”
3. Audit-Log: action = 'stripe_capture_failed'
charge.dispute.created

Tritt auf wenn Gast Chargeback einleitet
Aktion:
1. reservations.stripeAuthStatus = 'disputed'
2. guests.isHighRisk = 1
3. Admin-Alert: “Chargeback für Buchung [ID]”
4. Audit-Log: action = 'stripe_dispute_created'
charge.dispute.closed

Tritt auf wenn Chargeback abgeschlossen wird
Aktion:
1. reservations.stripeAuthStatus = 'dispute_closed'
2. Audit-Log: action = 'stripe_dispute_closed'

34.3 Idempotenz
Alle Stripe-Webhooks werden gegen processed_webhooks geprüft:
stripeEventId muss eindeutig sein (72h TTL)

Wenn bereits verarbeitet: 200 OK, keine Aktion

34.4 Capture-Flow (No-Show)
Wenn status = 'no_show' und stripeAuthStatus = 'authorized' :
1. stripe.paymentIntents.capture(paymentIntentId) aufrufen
2. Betrag = noShowFeeAmount × partySize (wenn noShowFeeType = 'per_person' )
3. Betrag = noShowFeeAmount (wenn noShowFeeType = 'per_booking' )
4. Bei Erfolg: stripeAuthStatus = 'captured'
5. Bei Fehler: stripeAuthStatus = 'capture_failed' , Admin-Alert

34.5 Release-Flow (Seated oder Storno durch Restaurant)
Wenn status = 'seated' oder Storno durch Restaurant:
1. stripe.paymentIntents.cancel(paymentIntentId) aufrufen
2. Bei Erfolg: stripeAuthStatus = 'released'
3. Bei Fehler: Retry nach 5 Min, max. 3 Versuche, dann Admin-Alert

Kapitel 36: Cron-Jobs — SQL-Logik
36.1 Job: noShowRelease
Frequenz: Jede Minuteuten
SQL-Logik:

-- Schritt 1: Alle Kandidaten finden
SELECT r.id, r.restaurantId, r.guestId, r.startTime,
res.noShowMinutes, res.noShowSmsEnabled, res.timezone
FROM reservations r
JOIN restaurants res ON r.restaurantId = res.id
WHERE r.status = 'confirmed'
AND r.startTime + INTERVAL res.noShowMinutes MINUTE < UTC_TIMESTAMP()
AND res.isActive = 1
FOR UPDATE;
-- Schritt 2: Für jede Buchung
UPDATE reservations
SET status = 'no_show', noShowAutoReleasedAt = UTC_TIMESTAMP(), updatedAt =
UTC_TIMESTAMP()
WHERE id = :reservationId AND status = 'confirmed';
-- Schritt 3: Gast-Counter erhöhen
UPDATE guests
SET noShowCount = noShowCount + 1, updatedAt = UTC_TIMESTAMP()
WHERE id = :guestId;
-- Schritt 4: Booking-Counter aktualisieren (No-Show zählt nicht als Buchung)
-- Kein Update nötig, No-Shows wurden bereits gezählt

Race Condition Schutz: FOR UPDATE + WHERE status = 'confirmed' im UPDATE verhindert
Doppelmarkierung.

36.2 Job: reminderSms
Frequenz: Täglich zur konfigurierten Uhrzeit (pro Restaurant unterschiedlich)
SQL-Logik:

-- Alle Buchungen für morgen finden (in Restaurant-Timezone)
SELECT r.id, r.restaurantId, r.guestId, r.startTime,
g.phone, g.language, g.smsConsentStatus,
res.reminderSmsEnabled, res.timezone
FROM reservations r
JOIN guests g ON r.guestId = g.id
JOIN restaurants res ON r.restaurantId = res.id
WHERE r.status = 'confirmed'
AND DATE(CONVERT_TZ(r.startTime, 'UTC', res.timezone)) =
DATE(DATE_ADD(CONVERT_TZ(UTC_TIMESTAMP(), 'UTC', res.timezone), INTERVAL 1
DAY))
AND r.reminderSmsSentAt IS NULL
AND res.reminderSmsEnabled = 1
AND g.smsConsentStatus = 1
AND g.smsOptedOut = 0;
-- Nach Versand:
UPDATE reservations SET reminderSmsSentAt = UTC_TIMESTAMP() WHERE id = :id;

Frequenz: Täglich um 04:00 UTC
SQL-Logik:

-- Schritt 1: Gäste anonymisieren (retentionDate überschritten, keine aktiven
Buchungen)
SELECT g.id FROM guests g
WHERE g.retentionDate < CURDATE()
AND g.deletedAt IS NULL
AND NOT EXISTS (
SELECT 1 FROM reservations r
WHERE r.guestId = g.id
AND r.status IN ('confirmed', 'seated', 'pending_authorization')

);

email = NULL,

deletedAt = UTC_TIMESTAMP()
WHERE id = :guestId;

const sig = req.headers['x-google-signature'];
const body = JSON.stringify(req.body);
const expectedSig = crypto
.createHmac('sha256', process.env.GOOGLE_BOOKING_SECRET)
.update(body)
.digest('hex');
if (!crypto.timingSafeEqual(
Buffer.from(sig),
Buffer.from(expectedSig)
)) {
return res.status(401).json({ error: 'Invalid signature' });
}

44.6 Idempotenz
Jede Google-Buchung hat eine eindeutige bookingId . Wenn dieselbe bookingId zweimal
gesendet wird (Retry), wird die Buchung nicht doppelt erstellt:

SELECT id FROM reservations
WHERE externalBookingId = :googleBookingId
AND restaurantId = :restaurantId
LIMIT 1;

Wenn gefunden: Bestätigung zurücksenden ohne neue Buchung zu erstellen.
